I keep coming back to this idea that the fundamental tension in AI products isn't about capability at all. It's about this weird loop where the model needs context to be useful, but users don't trust it enough to give it that context.

## The paradox

Here's the shape of it:

- Without memory, the assistant feels dumb. You re-introduce yourself every session.
- With memory, the assistant feels uncanny. It knows things you didn't consciously tell it.
- Users want the benefits of memory without the weight of it.

We keep trying to solve this with UI — settings toggles, "don't remember this" buttons, memory dashboards. I don't think it works. The problem isn't that users can't find the toggle. It's that the toggle itself surfaces the anxiety we're trying to dampen.

> Trust is the bottleneck, not capability. The best AI products build trust ramps, not capability mountains.

## Progressive delegation

The pattern I keep seeing work is something closer to **progressive delegation**: the model asks for small amounts of context, uses them well, and earns the right to ask for more. Each successful round lowers the cost of the next one.

A few concrete principles I'd apply:

1. **Never ask upfront for what you can infer mid-conversation.** If the user tells me their name in message three, don't make them tell me in a setup wizard.
2. **Summarize back before you save.** If you're going to remember something, show what you heard first. It's not a security layer — it's a trust ramp.
3. **Let memory decay, visibly.** Old context gets softer. Users notice this, and it maps to how actual human memory works.

## Why this is a design problem, not a model problem

The models are already plenty capable of remembering. The question is how to structure the remembering so it doesn't read as surveillance.

I don't have a clean answer. I do have a growing suspicion that most of the "AI UX" work I've seen ships capability without shipping the ramp, and wonders why adoption stalls at the third prompt.
